# astrbot_plugin_group_sender_allowlist

本目录为 **独立插件项目**，请整体复制到 AstrBot 的 `data/plugins/` 下使用。

插件根目录下的 **`logo.png`** 为 AstrBot / 插件市场展示图标；可自行替换同名 PNG。

本插件以 **[MIT License](LICENSE)** 开源。

---

## 一、`id_whitelist` 与 `group_rules`：易混场景

下面两种组合经常被混在一起理解，但 **AstrBot 白名单** 与 **本插件** 分工不同，请对照：

| 场景 | AstrBot `id_whitelist` | 本插件 `group_rules` | 群聊里谁能让机器人响应（简化表述） |
|------|------------------------|----------------------|--------------------------------------|
| **A** | **没有**该群的 `group_id`，也 **没有**能匹配到的会话 ID（如 `/sid` 给出的串） | 任意（空、有其它群规则、或没写该群都无所谓） | **通常谁也用不了**：消息过不了 AstrBot 白名单，**到不了本插件**。 |
| **B** | **有**该群 `group_id`（或能匹配的群会话串），消息能进 AstrBot | **整表为空**，或 **没有**以该群号开头的一行 `群号:...` | **全员都能用**：本插件 **只**对 `group_rules` **里出现过的群号**做发送者过滤；未出现的群 **等同于「本插件未管该群」**，**不会**因为「插件开着、规则为空」就自动全员禁止。 |

**要点**：若要 **模式 3**（整群进 AstrBot、群内只允许部分 QQ、且这部分人共享群级上下文），必须 **同时** 做到：（1）`id_whitelist` 放行该群；（2）`group_rules` 写 `群号:允许的QQ,...`。只完成（1）而不写（2），对应上表 **场景 B**，行为是全员可聊，不是「默认拒绝」。

## 二、三种模式（与 AstrBot 如何配合）

### 模式 1：开启会话隔离（`platform_settings.unique_session = true`）

| 维度 | 说明 |
|------|------|
| **群内上下文** | **不互通**。群聊下每人一条会话 ID（QQ 上常见为「用户ID_群号」形式），对话历史按人拆开。 |
| **AstrBot `id_whitelist`** | 要「只让部分群员进 AstrBot」，应把 **这些人的会话标识**（用 `/sid` 复制）写进白名单。**若只写裸 `group_id`，通常仍会放行该群全部成员**（与是否开会话隔离无关）。 |
| **私聊** | 无群号时往往不能只靠群号匹配，一般需用 **私聊的会话 ID**（`/sid` 复制）放行。 |
| **能做什么** | 每人独立上下文 + 可为不同群员分别配置会话 ID 放行。 |
| **不能做什么** | 被放行的多名群员之间 **不共享** 同一条 LLM 会话历史。 |

### 模式 2：关闭会话隔离（`unique_session = false`），仅 AstrBot 白名单

| 维度 | 说明 |
|------|------|
| **群内上下文** | **互通**。群聊会话按群维度，常见会话 ID 形如 `适配器实例名:GroupMessage:群号`（以 `/sid` 为准）。 |
| **AstrBot 白名单** | **群聊**：多填 **`group_id`** 或完整群会话串；**私聊**：填私聊会话 ID（`/sid`）。 |
| **不能做什么** | 群一旦用 `group_id` 放行，**无法在 AstrBot 层单独禁止群内部分成员**（人人进同一条会话）。 |

### 模式 3：关会话隔离 + AstrBot 放行群 + 本插件 `group_rules`

| 维度 | 说明 |
|------|------|
| **AstrBot** | `id_whitelist` 仍须包含 **`group_id`（或该群的会话串）**，否则消息进不来，本插件收不到。 |
| **本插件** | `群号:允许的QQ,...`；不在列表中的成员 **后续 Star / 默认 LLM 链路被拦**。 |
| **群内上下文** | `unique_session` 关 → 群内一条会话，**被放行的多名群员共享**同一条对话（未被拦下的发言才会进入后续逻辑与历史）。 |

若 **`group_rules` 里没有该群号行**（含整表为空），本插件 **不处理该群**，行为与 **模式 2** 一致；与「`id_whitelist` 里没写群号」的区别见 **一、易混场景**。

## 三、每种模式各举一例（虚构数据，便于套用自己环境）

以下假定 QQ 适配器 **`aiocqhttp`**，适配器实例在 AstrBot 里显示的 ID 为 **`我的机器人`**（以你机器上 `/sid` 为准）；群号 **`1000001`**；QQ 号 **`2000001`（小 A）**、**`2000002`（小 B）**、**`2000003`（小 C）**。

### 示例 A — 模式 1（会话隔离：按人、上下文不共享）

- **配置**：`platform_settings.unique_session = true`。  
- **白名单**：希望 **只有小 A、小 B** 能在群 `1000001` 里用机器人，**不要**小 C。  
  - **错误做法**：只把 `1000001` 写进 `id_whitelist` → **小 C 往往仍能通过 AstrBot 白名单**（整群放行）。  
  - **正确做法**：在 `id_whitelist` 中加入 **两条会话 ID**（用 `/sid` 实测复制）：  
    - `我的机器人:GroupMessage:2000001_1000001`  
    - `我的机器人:GroupMessage:2000002_1000001`  
- **结果**：小 A 与小 B 各有一套群内对话记忆；**互不共享**；小 C 过不了白名单（若未配置其会话 ID）。

### 示例 B — 模式 2（不关隔离：整群、上下文共享）

- **配置**：`platform_settings.unique_session = false`（或默认关）。  
- **白名单**：`id_whitelist` 中加入 `1000001`（或 `我的机器人:GroupMessage:1000001`）。  
- **结果**：小 A、小 B、小 C **任何人**在群内触发机器人时，都写入 **同一条** 群会话（如 `我的机器人:GroupMessage:1000001`，以 `/sid` 为准），**三人上下文完全互通**；**仅靠 AstrBot 白名单无法在群内单独禁止**小 C。

### 示例 C — 模式 3（关隔离 + 本插件：群放行、仅部分人、这部分人共享上下文）

- **配置**：`unique_session = false`。  
- **AstrBot `id_whitelist`**：仍加入 `1000001`（或该群的会话串），保证群消息能进管道。  
- **本插件 `group_rules`** 增加一行：  
  - `1000001:2000001,2000002`  
- **结果**：  
  - 小 A、小 B：通过白名单与本插件，使用 **同一条** 群级会话，**两人上下文互通**。  
  - 小 C：通过 AstrBot 白名单，但被本插件拦截，**不进入**后续插件与默认 LLM，**不会往这条共享群会话里续写**。  
- **私聊**：例如只允许 QQ `3000001` 私聊，仍在 `id_whitelist` 填其私聊会话 ID（如 `我的机器人:FriendMessage:3000001`，以 `/sid` 为准），**与上面群规则独立**。

### 示例 D — 仍是模式 3：多群、每群不同「谁有权」、群与群之间上下文天然隔离

**需求（与你的描述一致）：**

- 群 **K**：成员 A、B、C 都能和 AstrBot 聊，且 **三人共享** 群 K 里的那一条对话上下文。  
- 群 **J**：同样是 A、B、C 三个人在群里，但只希望 **A、B** 共享群 J 的上下文；**C 在群 J 无权访问** AstrBot。  
- 群 K 与群 J 的上下文 **不应混成一条**（各群各有一份记忆）。

**这是否是「第四种模式」？**  
**不是。** 仍是 **模式 3**：`unique_session` 关 + AstrBot 白名单放行相关群 + 本插件按 **群号** 分别配置允许列表。关会话隔离时，不同群对应不同会话 ID（随群号区分），因此 **K 与 J 各有一套上下文**，不会把两群聊记混。

**配置示例**（群号、QQ 请换成真实值）：

1. **`platform_settings.unique_session = false`**  
2. **AstrBot `id_whitelist`**：同时包含 **`K`**、**`J`** 的群号（或各自的群会话串），使两个群的消息都能进 AstrBot。  
3. **本插件 `group_rules`** 写 **两行**（每群一条规则）：  
   - `K:A的QQ,B的QQ,C的QQ`  
   - `J:A的QQ,B的QQ`（**故意不写 C**）

**结果：**

- 在 **群 K**：A、B、C 均通过插件校验，读写 **群 K** 对应的那条共享会话，**三人上下文互通**。  
- 在 **群 J**：A、B 通过；C 被本插件拦截，**不能**驱动后续链路，**不参与**群 J 的共享上下文。  
- **群 K 与群 J**：各群各一条会话，**两套上下文互不合并**。

若某群 **未**出现在 `group_rules` 里，本插件不拦截该群，行为会退化成「仅 AstrBot 白名单」；多群场景下请 **每个要精细管控的群都写一行** `group_rules`。

## 四、本插件在模式 3 中的角色

- **不替代** AstrBot 白名单：只做 **群内的发送者二次校验**。  
- 通过 **高优先级 Handler + 裁剪后续 Star**，尽量让非白名单成员 **不触发** 其它插件与默认 LLM（与具体第三方插件优先级有关，详见「限制」）。  
- **每次按 `group_rules` 拦截**都会在 AstrBot 日志打一条 **INFO**，前缀为 **`[astrbot_plugin_group_sender_allowlist]`**，写明插件展示名、`clamp_unauthorized_group_sender`、配置项 **`group_rules`**、群号、发送者及该群允许人数（便于区分「空列表全员拒」与「列表里没他」）。  
- 详细配置项见 **`_conf_schema.json`** / WebUI。

---

## 五、配置格式（`group_rules`）

每项一行：**`群号:允许的QQ号,QQ号,...`**

- 仅写 **`1000001:`**（冒号后留空）表示该群允许列表为空，非管理员豁免下 **全员拦截**（仍须先把该群放进 AstrBot 白名单）。  
- **未在 `group_rules` 中出现过的群号**：本插件 **不处理**，等同仅 AstrBot 白名单（见 **一、易混场景** 表 **场景 B**）。

## 六、常见问题（误以为没生效）

1. **`group_rules` 为空 / 没写该群，但机器人仍回复** → **预期行为**：`id_whitelist` 里若已写群号，AstrBot 会放行全员；本插件 **只对 `group_rules` 里写过的群号**生效。与「`id_whitelist` 里根本没写该群」不同，后者通常是 **谁也聊不了**（见 **一、易混场景**）。  
2. **测试号在 AstrBot `admins_id` 且开启「管理员豁免」** → 该号在受管群内始终放行。  
3. **QQ 号以日志、`/sid` 里的全数字为准**。  
4. 若仍有第三方插件在拦截前执行，可将 `main.py` 中的 **`_HANDLER_PRIORITY`** 再调大。

---

## 七、限制

- 依赖适配器正确提供 `get_group_id()`、发送者信息（本插件对 `user_id` 为 **int** 的情况做了兼容读取，与 AstrBot 核心 `get_sender_id()` 仅返回 str 的行为互补）。若仍无法解析发送者 ID，为安全起见不拦截该条（并可在开启 `log_blocked` 时打 WARNING）。  
- 仅处理 **群聊**；私聊访问控制仍以 AstrBot 白名单为主。  
- 无法保证拦截 **其它插件内部自行发起的 LLM 调用**（与 AstrBot 对 `call_llm` 等机制一致）。  
- 修改 `group_rules` 后通常需 **重载插件** 后才会重新解析（`initialize` 时加载）。
